# 레퍼런스 라이브러리 — 디자인 사례와 원칙

외관을 베끼는 용도가 아니다. 방향을 잡을 때 "이 상황에서 그들이라면 어떤 원칙을 적용했을까"를
빌려오는 용도다. Deadline Flow에 적용 가능한 원칙만 추렸다.

## Apple Human Interface Guidelines (기본기 — 가장 자주 참조)

- **Clarity**: 모든 화면에서 "사용자가 지금 알아야 할 한 가지"가 즉시 읽혀야 한다.
  이 앱에서는 "남은 시간이 얼마나 되나". 그것보다 먼저 눈에 들어오는 게 있으면 위계 오류.
- **Deference**: UI는 콘텐츠에 양보한다. 크롬(보더, 배경, 장식)은 콘텐츠(태스크, 시간)를
  이기면 안 된다. 장식이 눈에 띄면 이미 과한 것.
- **피드백은 즉각적이고 비례적으로**: 클릭에는 즉시 반응(pressed scale), 상태 변화에는
  부드러운 전이. 작은 액션에 큰 애니메이션을 붙이지 않는다.
- **터치/클릭 타겟**: 최소 44pt에서 출발하되 데스크톱 밀도에서는 32px까지 허용.
  시각적 크기와 히트 영역을 분리해도 된다 (작은 아이콘 + 넓은 padding).
- **시스템의 결을 따른다**: macOS/Windows 데스크톱 앱이므로 스크롤바, 포커스 링,
  reduced-motion 같은 OS 관례를 존중한다.

## Stripe (정보 밀도와 신뢰감)

- 숫자·데이터가 많은 UI에서 신뢰감은 **정렬과 tabular-nums**에서 나온다.
  시간·통계 화면에서 숫자 끝자리가 흔들리면 아마추어 티가 난다.
- 색은 아껴 쓰되, 쓰는 곳에서는 명확하게 — 파스텔로 뭉개지 않고 한 번에 읽히는 대비.
- 문서적 위계: 큰 제목 아래 조용한 본문, 섹션 사이 넉넉한 여백. 박스로 나누기보다
  여백과 타이포로 나눈다.

## Linear (절제된 프로덕트 UI — 이 앱과 가장 결이 가까움)

- 거의 무채색 화면에 포인트 컬러는 상태 표시로만. 화면이 심심해 보여도 버틴다 —
  콘텐츠가 들어오면 그 절제가 품질로 읽힌다.
- 인터랙션 디테일이 품질감의 대부분: hover 배경, pressed scale, 보조 컨트롤의
  hover 시 노출(ghost 패턴), 150ms 전이. 정적 스타일보다 여기 투자한다.
- 리스트 행은 조용하게, 선택된 행만 확실하게. 행마다 아이콘·뱃지를 붙이지 않는다.

## Swiss / International Typographic Style (레이아웃 골격)

- **그리드가 먼저다**: 요소를 배치하고 정렬을 맞추는 게 아니라, 정렬선(그리드)을 먼저
  정하고 요소를 얹는다. 좌측 패널·우측 패널 각각 하나의 좌측 정렬선.
- **비대칭 균형**: 좌우 패널이 폭·밀도가 달라도 시각적 무게가 균형이면 된다.
  큰 타이머(시각적 무게 큼) vs 촘촘한 리스트(정보 밀도 큼) — 지금 구조가 이 원칙이다.
- **타이포그래피가 곧 디자인**: 장식 없이 크기·굵기·색·간격만으로 완성한다는 태도.

## Bauhaus (형태의 근거)

- 형태는 기능을 따른다 — radius, 원형, 크기는 전부 기능적 이유가 있어야 한다.
  원형 버튼은 "하나의 액션"이라서 원형이고, 카드가 둥근 건 창 안의 떠 있는 표면이라서다.
- 기하학적 순수성: 원은 정원으로, 정렬은 수학적으로. "대충 둥근" 것이 제일 나쁘다.

## 안티 레퍼런스 (따라하지 않는 것)

- 대시보드 템플릿 스타일 (그라데이션 카드, 아이콘+숫자 타일 그리드, 도넛차트 남발)
- 모바일 앱 관례의 무비판적 이식 (하단 탭바, 큰 CTA 버튼) — 이건 데스크톱 위젯 앱이다
- "friendly" 과잉 (이모지, 말풍선 톤, 통통 튀는 애니메이션 전면 배치)
