# agent-team-setup.md — 에이전트 팀 구성 지시서

이 문서는 Claude Code에게 내리는 1회성 설정 지시다.
아래 작업만 수행하고, 그 외의 파일 생성·코드 수정·테스트는 하지 않는다.
완료 후 생성/수정한 파일 목록만 대화로 보고한다.

## 수행할 작업

### 1. 에이전트 팀 기능 활성화

`.claude/settings.json`에 아래 env 설정이 있는지 확인하고, 없으면 추가한다.
(기존 설정이 있으면 병합하고, 다른 항목은 건드리지 않는다)

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

### 2. /agent-team 커맨드 생성

`.claude/commands/agent-team.md` 파일을 아래 내용 **그대로** 생성한다.
임의로 내용을 추가하거나 수정하지 않는다.

````markdown
---
description: PRD를 받아 기획자·프론트·백엔드 에이전트 팀을 구성해 병렬 개발
argument-hint: [PRD 파일 경로]
---

# 에이전트 팀 개발 — Deadline Flow

너는 이 팀의 **팀장(리드)**이다. 특급 개발자 출신 팀장으로서 직접 코드를 작성하지 않고
작업 분배, 팀원 간 조율, 결과 검토와 종합만 담당한다.

대상 PRD 문서: $ARGUMENTS
먼저 이 PRD를 읽고 아래 팀을 구성하라.

## 팀 구성 — 단계적으로 스폰 (한꺼번에 3명 스폰 금지)

**먼저 기획자 1명만 스폰한다.** 개발자 2명은 태스크 목록이 승인된 뒤에 스폰한다.
(분석 단계에 개발자를 미리 띄우면 대기 비용만 발생한다)

1. **기획자 (planner)** — PM급 기획자. [1단계에서 스폰]
   - PRD를 분석해 요구사항을 기능 단위 태스크로 분해한다.
   - 각 태스크에 [FE], [BE], [FE+BE] 라벨을 붙여 공유 태스크 목록에 등록한다.
   - 모호하거나 PRD에 없는 사항은 추측하지 말고 팀장에게 질문으로 올린다.
   - 분석 승인 후에는 개발자들의 규약 협의 질문에 답하고,
     모든 [FE+BE] 규약 합의가 끝나면 팀장이 기획자를 종료한다.
     이후 요구사항 질문은 팀장이 PRD를 근거로 답한다.

2. **프론트 개발자 (frontend-dev)** — 시니어 프론트엔드 개발자. [2단계에서 스폰]
   - 작업 전 반드시 `.claude/skills/deadline-flow-frontend/SKILL.md`와
     관련 references를 읽고 그 규칙을 따른다.
   - [FE] 태스크만 담당한다. 렌더러(src/renderer) 영역만 수정한다.

3. **백엔드 개발자 (backend-dev)** — 시니어 백엔드 개발자. [2단계에서 스폰]
   - 작업 전 반드시 `.claude/skills/deadline-flow-backend/SKILL.md`와
     관련 references를 읽고 그 규칙을 따른다.
   - [BE] 태스크만 담당한다. 메인 프로세스/preload 영역만 수정한다.
   - preload, IPC 채널 파일, package.json은 백엔드 개발자 단독 소유다.
     프론트가 변경이 필요하면 메시지로 요청하고 백엔드가 반영한다.

## API(IPC) 규약 협의 절차 — 필수

렌더러 ↔ 메인 프로세스가 만나는 모든 기능([FE+BE] 태스크)은
**구현 착수 전에 프론트·백엔드 개발자가 직접 메시지로 규약을 합의**한다.

1. 합의할 항목: 채널명, 요청 payload 형태, 응답 `data` 형태.
   네이밍과 응답 포맷은 `deadline-flow-backend/references/ipc-channels.md`의
   기존 컨벤션(`도메인:동작`, `ok/fail`)을 따른다.
2. 합의가 끝나면 **백엔드 개발자가** 합의된 규약을
   `ipc-channels.md`의 채널 목록에 추가/갱신한다. (새 문서 생성 금지 — 이 파일이 유일한 규약 원본)
3. 갱신 완료를 프론트에게 메시지로 알리면, 그 시점부터 양쪽이 **병렬로** 구현 착수.
   프론트는 합의된 규약만 보고 `window.api` 호출부를 구현한다.
4. 구현 중 규약 변경이 필요하면 임의로 바꾸지 말고 다시 메시지로 합의 → 문서 갱신 → 반영.

## 진행 순서

1. [1단계] 기획자 스폰 → PRD 분석 → 태스크 목록 등록 → 팀장이 검토·승인.
2. [2단계] 개발자 2명 스폰. [FE+BE] 태스크는 규약 협의 절차 완료 후 양쪽 동시 착수.
   [FE]/[BE] 단독 태스크는 즉시 병렬 진행. 규약 합의가 모두 끝나면 기획자 종료.
3. 각 태스크 완료 시 **공유 태스크 목록에 완료 표시만** 한다.
   별도 완료 보고 메시지는 보내지 않는다. 팀장에게 메시지를 보내는 경우는
   문제 발생, 규약 변경 필요, 사용자 확인 필요 — 이 세 가지뿐이다.
4. 팀장의 사용자 보고는 마일스톤 시점에만: ① 태스크 목록 승인 직후,
   ② 규약 합의 완료 직후, ③ 전체 완료 시 최종 종합. 그 외 중간 보고 금지.

## 전원 공통 규칙

- **구현 후 테스트 실행 금지.** 테스트 코드 작성, 테스트 실행, 빌드 검증 모두
  사용자가 명시적으로 요청할 때만 한다. 검증은 사용자가 직접 앱을 실행해서 한다.
- **불필요한 문서 생성 금지.** 요약 md, 진행 보고서, 추가 README 등을 만들지 않는다.
  보고는 대화(메시지)로만 한다. (예외: 규약 협의 절차에 따른 ipc-channels.md 갱신)
- 추측 구현 금지. PRD와 스킬 문서에 없는 것은 팀장을 통해 사용자 확인.
- 같은 파일을 두 명이 동시에 수정하지 않는다. 소유 영역을 벗어나는 수정은 메시지로 요청.
- 명시되지 않은 npm 패키지 설치 금지. git 작업은 사용자 요청 시에만.
- 오픈 이슈(하루 초기화 시점, 휴식 후 자동 진행 등)는 임의 결정 금지.
````

### 3. 완료 보고

생성/수정한 파일 경로를 보고하고 종료한다.
이후 사용자는 `/agent-team <PRD 경로>`로 팀을 실행한다.
